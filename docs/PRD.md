# [PRD] 실시간 예약 및 스케줄링 플랫폼

## 1. 서비스 개요 (Service Overview)

**1.1 서비스명:** 빈자리(Binjari.com)  
**1.2 서비스 타입:** 반응형 웹 서비스 (Web Service)  
**1.3 서비스 핵심 요약:** 비어있는 시간을 가장 확실하게 채우는 방법, 동시성 제어 기반의 실시간 예약 스케줄링 플랫폼  

**1.4 목적**
1. **기술적 측면: 완벽한 예약 무결성 보장 (Technical Reliability)**
   - 다수의 사용자가 동일한 시간대(Slot)에 동시 접속하여 예약을 시도할 때 발생할 수 있는 '중복 예약(Race Condition)' 문제를 원천적으로 차단합니다.
   - 신뢰성 있는 동시성 제어 로직을 통해 데이터의 정합성을 유지하며, 안정적인 실시간 시간대 점유 프로세스를 구현하는 것을 최우선 목표로 합니다.
2. **사용자 측면: 직관적이고 마찰 없는 예약 경험 제공 (User Experience)**
   - 복잡한 절차 없이, 사용자가 원하는 날짜와 비어있는 시간대를 한눈에 파악할 수 있는 직관적인 캘린더 UI를 제공합니다.
   - 실시간 상태 업데이트를 통해 예약 가능 여부를 즉각적으로 인지하게 하여 사용자 편의성을 극대화합니다.
3. **운영자 측면: 스케줄 관리의 완전 자동화 (Operational Efficiency)**
   - 운영자(호스트)가 수동으로 예약을 확인하고 일정을 조율하는 리소스를 최소화합니다.
   - 가용 시간 등록부터 예약 확정까지의 흐름을 시스템화하여, 운영자는 본연의 서비스 제공에만 집중할 수 있는 자동화된 관리 환경을 제공합니다.

## 2. 제품 범위 및 정책 (Scope & Policies)

**2.1 타겟 사용자 (Target Audience)**
- **예약자:** 특정 서비스(상담, 레슨, 회의실 등)를 이용하기 위해 빈 시간을 확인하고 예약하려는 개인.
- **호스트:** 자신의 가능한 시간을 등록하고 예약을 관리하는 운영자.

**2.2 MVP 범위 (Minimum Viable Product Scope)**
1. **사용자(예약자) MVP 기능:**
   - 회원가입 / 로그인 (이메일, 구글 OAuth)
   - 월 단위 캘린더 조회
   - 예약 가능한 슬롯 조회
   - 슬롯 선택 후 예약 요청
   - 본인 예약 내역 조회
   - 예약 취소
   - 예약 상태 변경 실시간 알림 수신
2. **호스트 MVP 기능:**
   - 운영 시간 및 예외 일정 설정
   - 슬롯 일괄 생성 및 관리
   - 예약 요청 조회
   - 예약 승인 / 거절 처리
   - 예약 상태 변경 실시간 알림 발송
   - 기본 통계 대시보드 조회(일간/주간 예약 건수, 승인율, 인기 시간대)
3. **시스템 MVP 범위:**
   - 동시성 제어 기반 중복 예약 방지
   - Redis 기반 임시 선점
   - Idempotency-Key 기반 중복 요청 방지
   - UTC 저장 및 호스트 타임존 기준 정책 검증
   - WebSocket 기반 실시간 알림

**2.3 MVP 제외 범위 (Out of Scope)**
- 결제 및 환불 기능
- SMS / 카카오 알림톡 발송
- 외부 캘린더 연동 (예: Google Calendar, Outlook Calendar)
- 다중 호스트 조직/팀 관리 기능
- 패키지 상품 / 정기권 / 반복 예약 기능
- 리뷰 / 평점 기능
- 쿠폰 / 프로모션 / 정산 기능
- 관리자(Admin) 전용 백오피스 **UI·화면의 전부 구현** (역할·정책 정의는 [`docs/인증인가.md`](./인증인가.md) §2.1 참고)
- AI 추천 슬롯 / 자동 스케줄 최적화 기능

**2.4 핵심 비즈니스 정책 (Business Rules)**
1. **예약 가능 정책:**
   - 예약은 슬롯 시작 시각 기준 최소 2시간 전까지만 가능하도록 합니다.
   - 예약은 오늘 기준 최대 30일 이내의 슬롯까지만 허용합니다.
   - 슬롯 기본 단위는 30분으로 하며, 호스트는 서비스 유형에 따라 60분 단위로 설정할 수 있습니다.
   - 호스트는 정규 운영 시간 내에 고정적인 휴게 시간(Break Time, 예: 점심시간 12:00~13:00)을 설정할 수 있으며, 해당 시간대에는 슬롯이 생성되지 않도록 처리합니다.
   - 호스트는 예약 간 버퍼 타임(Buffer Time)을 설정할 수 있으며, 버퍼 시간 동안은 추가 예약이 불가능합니다.
2. **승인 및 상태 정책:**
   - 호스트는 예약 처리 방식을 자동 승인(Auto Approval) 또는 수동 승인(Manual Approval)으로 선택할 수 있습니다.
   - 수동 승인 방식에서는 예약 요청 생성 시 상태를 `PENDING`으로 저장하고, 호스트 승인 시 `CONFIRMED`, 거절 시 `REJECTED`로 변경합니다.
   - 자동 승인 방식에서는 예약 생성과 동시에 `CONFIRMED` 상태로 확정됩니다.
3. **사용자 예약 제한 정책:**
   - 사용자는 동시에 보유할 수 있는 활성 예약 건수를 제한할 수 있습니다. (예: `PENDING` + `CONFIRMED` 상태 예약 합산 최대 3건)
   - 동일 사용자가 동일 시간대에 중복 예약을 시도하는 경우 차단합니다. 
4. **취소 및 노쇼 정책:**
   - 예약 취소는 슬롯 시작 시각 기준 24시간 전까지만 허용합니다.
   - 슬롯 시작 시각 이후 미방문 / 미이용 상태가 확인되면 `NO_SHOW` 상태로 전환할 수 있습니다.
   - `REJECTED`, `CANCELLED`, `NO_SHOW` 상태가 된 예약은 통계 및 이력에서 구분 관리합니다.
5. **시간 정책:**
   - 모든 저장 시각은 UTC 기준으로 관리합니다.
   - 예약 가능 여부, 취소 가능 시점, 슬롯 생성 규칙은 호스트의 운영 시간대(`host_timezone`) 기준으로 판단합니다.
   - 사용자 화면에서는 접속 기기의 로컬 타임존으로 변환해 표시합니다.

**2.5 역할·권한 정책 (요약)**  
Guest(비로그인 조회만) · Member(로그인·예약·내 예약, 회원가입 자동 승인, 호스트 운영 불가) · Host(`HOST`, 예약 페이지 운영·로그인 후 기본 진입은 예약 페이지 목록) · Admin(`ADMIN`, 가입 심사·계정 조치 등). 상세·API 매핑은 [`docs/인증인가.md`](./인증인가.md) §2.1을 따른다.

## 3. 사용자(예약자) 기능 명세

### 3.1 사용자 인증 및 인가 (Authentication & Authorization)
- **기능 요약:** 이메일 기반의 자체 회원가입/로그인 및 OAuth 2.0 기반 구글 소셜 로그인 지원.
- **기술적 구현 방안:**
  - **보안 및 암호화:** 사용자의 비밀번호는 단방향 해시 알고리즘(Bcrypt 등)을 사용하여 안전하게 데이터베이스에 저장합니다.
  - **JWT 토큰 기반 인증:** 로그인 성공 시 Access Token(단기)을 응답으로 반환하고, Refresh Token(장기)은 HttpOnly Secure Cookie로 발급하여 무상태(Stateless) 기반의 안전한 세션 관리를 구현합니다.
  - **OAuth 2.0 연동:** 구글 로그인 시 Authorization Code Grant 방식을 사용하여 인가 코드를 받고, 이를 백엔드에서 구글 API와 통신하여 사용자 정보를 안전하게 획득 및 검증합니다.

### 3.2 캘린더 뷰어 (Calendar & Slot Visualization)
- **기능 요약:** 호스트가 개설한 예약 가능 슬롯을 사용자의 브라우저 상에 날짜/시간별로 직관적으로 렌더링.
- **기술적 구현 방안:**
  - **데이터 페칭 최적화:** 캘린더 렌더링 시 전체 데이터를 가져오지 않고, 사용자가 보고 있는 '해당 월(Month)'의 시작일과 종료일을 쿼리 파라미터로 전달하여 필요한 구간의 슬롯 데이터만 조회합니다.
  - **Timezone 정합성 보장:** 데이터베이스에는 모든 예약 시간을 국제 표준시(UTC) 기준으로 저장하고, 프론트엔드 단에서 접속한 사용자의 로컬 타임존(Local Timezone)으로 변환하여 렌더링하는 로직을 적용합니다.
  - 호스트별 운영 기준 시간대(`host_timezone`)를 별도로 저장하여, 슬롯 생성 및 예약 가능 여부 판단은 호스트 기준 시간대로 처리하고, 최종 표시만 사용자 로컬 타임존으로 변환합니다.

### 3.3 실시간 예약 신청 (Real-time Booking Request) - ⭐️ 핵심 과제
- **기능 요약:** 캘린더에서 빈 슬롯을 선택하고 예약을 확정하는 프로세스.
- **기술적 구현 방안:**
  - **동시성 제어 (Concurrency Control):** 동일한 슬롯에 다수의 트래픽이 몰릴 경우, 사용자 경험 측면에서는 Redis 기반 임시 선점(Temporary Hold)으로 짧은 시간의 경합을 완화하고, 최종 예약 무결성은 데이터베이스 트랜잭션과 고유 제약조건(Unique Constraint)을 통해 보장합니다.
  - **트랜잭션(Transaction) 관리:** 예약 데이터 생성과 슬롯 상태 변경은 하나의 작업 단위로 묶어 처리하며, 중간 실패 시 전체 롤백(Rollback)됩니다. 결제 정보 저장은 MVP 제외 범위이며, 최종 확정 시에는 슬롯 레코드에 대해 배타적 잠금을 획득하여 동일 슬롯에 대한 경쟁 요청이 직렬화되도록 합니다.

### 3.4 예약 현황 조회 (Booking Status Management)
- **기능 요약:** 사용자가 자신의 예약 히스토리 및 현재 상태(대기/확정/취소)를 확인하는 대시보드.
- **기술적 구현 방안:**
  - **인덱스(Index) 최적화:** 예약 테이블에서 특정 사용자의 ID와 예약 상태값을 기준으로 조건 검색이 빈번하게 일어나므로, 해당 컬럼들에 적절한 복합 인덱스를 설정하여 조회 성능(Query Performance)을 높입니다.
  - **페이징/무한 스크롤:** 예약 내역이 많아질 경우를 대비해 오프셋(Offset) 방식 또는 커서(Cursor) 방식의 페이지네이션 API를 구현하여 네트워크 부하를 줄입니다.

### 3.5 실시간 알림 시스템 (Real-time Notification via WebSockets)
- **기능 요약:** 호스트가 예약을 확정하거나 사용자가 예약을 취소할 때, 새로고침 없이 즉각적으로 UI 팝업(Toast) 알림 제공.
- **기술적 구현 방안:**
  - **양방향 통신 구현:** HTTP의 단방향 통신 한계를 극복하기 위해 WebSocket 프로토콜을 사용합니다.
  - **실시간 연결 관리:** 단일 서버 환경에서는 메모리 기반 Connection Manager를 사용할 수 있으나, 실제 운영 환경에서는 다중 인스턴스(Multi-instance) 확장을 고려하여 Redis Pub/Sub 또는 메시지 브로커를 이용해 서버 간 이벤트를 전파합니다.
  - 특정 이벤트 발생 시 타겟 사용자(예약자 본인 또는 해당 호스트)에게만 비동기적으로 메시지를 전달하며, 알림 누락 시 재조회 API를 통해 최신 상태를 복구할 수 있도록 설계합니다.

## 4. 호스트(운영자) 기능 명세

### 4.1 가용 시간 설정 (Time Slot Generation & Management)
- **기능 요약:** 호스트가 자신의 운영 정책(예: 월~금 09:00~18:00, 30분 단위)에 맞춰 예약 가능한 슬롯을 일괄 생성하고, 특정 날짜의 휴무일 등 예외 일정을 관리.
- **기술적 구현 방안:**
  - **데이터 모델링:** 기본 요일 패턴을 저장하는 규칙(Rule) 테이블과, 특정 날짜의 휴무나 단축 근무를 덮어씌우는 예외(Override) 테이블을 분리하여 유연하고 확장성 있는 스케줄링 스키마를 설계합니다.
  - **Bulk Insert 최적화:** 선택한 기간(예: 1개월치)의 30분 단위 슬롯을 생성할 때, 데이터베이스에 수백 개의 쿼리를 개별적으로 날리지 않습니다. ORM(SQLModel)의 일괄 삽입(Bulk Insert) 기능이나 executemany를 활용해 트랜잭션 오버헤드를 최소화하고 서버 성능을 확보합니다.
  - 슬롯 생성 시 기준 시간대는 호스트의 운영 시간대(`host_timezone`)를 따르며, 생성된 결과는 UTC 기준으로 저장합니다.
  - buffer_duration_mins는 슬롯 생성 시 각 예약 블록 사이의 간격으로 반영합니다. 예를 들어 30분 슬롯 + 10분 버퍼이면 다음 예약 가능 시작 시각은 직전 슬롯 종료 10분 후가 됩니다.

### 4.2 예약 승인/거절 및 상태 관리 (Booking State Management)
- **기능 요약:** 대기 상태(`PENDING`)의 예약 요청을 확인하고 승인(`CONFIRMED`) 또는 거절(`REJECTED`) 상태로 변경하며, 그에 따른 슬롯 상태를 동기화.
- **기술적 구현 방안:**
  - **상태 머신 (State Machine) 패턴 적용:** 예약의 라이프사이클을 명확하게 정의하고, 유효하지 않은 상태 변경 요청을 백엔드 비즈니스 로직에서 철저히 검증하여 방어합니다.
  - **원자적 상태 롤백 (Atomic State Rollback):** 호스트가 예약을 '거절'할 경우, 예약 데이터의 상태 변경과 동시에 해당 시간대 슬롯의 상태를 다시 `OPEN`(예약 가능)으로 되돌리는 두 작업을 단일 트랜잭션으로 묶어 데이터 불일치를 원천 차단합니다.
  - **이벤트 기반 통신:** 상태 변경이 데이터베이스에 성공적으로 커밋(Commit)된 이후에만 웹소켓 이벤트를 발행하여 상태 불일치를 방지합니다.

### 4.3 운영 대시보드 및 통계 (Dashboard & Analytics)
- **기능 요약:** 일간/주간 예약 건수, 승인율, 인기 시간대 등의 통계 데이터를 차트로 시각화.
- **기술적 구현 방안:**
  - **복잡한 집계 쿼리 최적화:** 예약 날짜(`date`) 및 호스트 ID 컬럼에 적절한 데이터베이스 인덱스를 적용합니다.
  - **데이터 캐싱 (Caching):** 데이터베이스 조회 결과를 Redis 등에 일정 시간 캐싱(TTL 설정)하여 API 응답 시간을 대폭 개선합니다.
  - **프론트엔드 렌더링 최적화:** `useMemo` 훅을 활용하여 가공된 통계 데이터를 메모이제이션 처리하여 불필요한 리렌더링을 방지합니다.

## 5. 시스템 핵심 로직 (Backend Core Logic)

### 5.1 동시성 제어 (Concurrency Handling) 및 트랜잭션 관리
> **목표:** 동일한 슬롯에 2명 이상 예약 시 중복 예약을 완벽히 방지.
- 최종 예약 무결성은 Redis 락만으로 보장하지 않고, 데이터베이스 트랜잭션과 고유 제약조건을 통해 보장합니다.
- 슬롯 확정 시점에는 데이터베이스 차원에서 특정 시간대(Slot) 레코드에 대해 `SELECT ... FOR UPDATE`를 수행하여 동시 수정 충돌을 방지하고, 직렬화되도록 처리합니다.
- Redis는 임시 선점과 빠른 상태 반영을 위한 보조 계층으로 사용하며, 최종 확정 여부의 단일 진실 공급원(Source of Truth)은 PostgreSQL로 정의합니다.
- Pessimistic Lock 사용 시 발생할 수 있는 교착 상태(Deadlock)를 예방하기 위해, 여러 슬롯이나 자원을 점유할 경우 항상 일관된 순서(예: 슬롯 ID 오름차순)로 락을 획득하도록 쿼리 순서를 제어합니다.

### 5.2 시간대 정합성 보장 (Timezone Management)
> **목표:** 서버(UTC)와 클라이언트(Local) 간의 시간차 계산 및 저장 오차 방지.
- **UTC 단일화 원칙:** 백엔드 서버 시스템 시간, 데이터베이스 타임스탬프, API 통신 규격을 모두 UTC 기준의 ISO 8601 포맷으로 통일합니다.
- 백엔드는 호스트별 운영 시간대(`host_timezone`)를 기반으로 정책 검증을 처리합니다.
- 프론트엔드는 사용자 화면에 표시할 때만 접속한 기기의 로컬 타임존으로 변환하여, 일광 절약 시간제(DST) 등 오류 가능성을 줄입니다.

### 5.3 임시 선점 및 타임아웃 처리 (Temporary Hold & TTL)
> **목표:** 사용자가 예약을 진행하는 과정(예: 폼 작성, 요청 사항 입력) 중에 다른 사람이 같은 자리를 채워버리면 안 됩니다.
- 예약 버튼을 누르는 순간, Redis를 활용해 해당 슬롯에 '임시 락(Lock)'을 적용합니다 (TTL: 5분).
- Redis의 TTL(Time-To-Live) 기능을 사용하여 5분 뒤에 자동으로 락이 해제되도록 설정합니다. 
- 단, Redis 임시 선점은 사용자 경험 개선과 경합 완화 목적이며, 최종 예약 확정 시에는 반드시 데이터베이스 트랜잭션 내 검증을 다시 수행합니다.
- 5분 안에 예약 확정 API가 호출되면 DB에 최종 반영하고 Redis 락을 해제하며, 시간 초과 시 다른 사용자가 다시 예약할 수 있도록 처리합니다.

### 5.4 대량 데이터 조회 성능 개선 (N+1 Query Optimization)
> **목표:** ORM 지연 로딩(Lazy Loading)으로 인한 성능 저하 방지.
- 캘린더 조회 API 호출 시, 연관된 테이블 데이터를 한 번의 쿼리로 가져오도록 즉시 로딩(`Eager Loading`, `joinedload` 등)을 전략적으로 적용합니다.
- 데이터베이스 실행 계획(Explain)을 분석하여 불필요한 조인(Join)을 줄이고 응답 지연 원인을 문서화합니다.

### 5.5 API 멱등성 보장 (API Idempotency)
> **목표:** 네트워크 지연 시 중복 예약 요청 방지.
- 클라이언트가 예약 요청 시 고유한 `Idempotency-Key`(UUID 등)를 헤더에 포함합니다.
- 백엔드는 이 키를 Redis나 DB에 저장하고, 중복 요청이 들어오면 서버 로직을 다시 실행하지 않고 이전 성공 응답 값을 그대로 반환합니다.

### 5.6 실시간 알림 아키텍처 보강
> **목표:** 멀티 서버 환경에서의 웹소켓 알림 누락 방지.
- WebSocket 연결은 서버별로 관리하되, 이벤트는 Redis Pub/Sub 또는 메시지 브로커를 통해 전체 인스턴스에 전파합니다.
- 알림 실패 시 클라이언트는 재접속 후 최신 예약 상태를 조회하여 화면 정합성을 회복합니다.

## 6. 기술 스택 제안 (Technical Stack)
- **Frontend:** React (Vite), Tailwind CSS (스타일링), React Query (서버 상태 관리)
- **Backend:** FastAPI (Python의 비동기 성능 활용)
- **Database:** PostgreSQL (관계형 데이터 처리) + SQLModel (FastAPI와 찰떡궁합 ORM)
- **Caching/Locking:** Redis (실시간 예약 임시 선점, Idempotency-Key 저장, 캐싱), Redis Pub/Sub (알림 이벤트 전파)

## 7. API 문서화 및 에러 핸들링 (API Design & Error Handling)
- **자동화된 API 문서:** FastAPI에 내장된 OpenAPI(Swagger UI)를 활용하여 코드와 동기화된 인터랙티브 API 문서를 제공합니다.
- **일관된 예외 처리:** 비즈니스 로직 에러를 중앙화된 Exception Handler에서 가로채어 규격화된 JSON 포맷(예: `{"error_code": "SLOT_ALREADY_BOOKED", "message": "이미 예약된 시간입니다."}`)으로 응답합니다.

## 8. 테스트 전략 (Testing & QA Strategy)
- **백엔드 테스트 (pytest):** FastAPI의 TestClient와 pytest를 활용하여 주요 API 엔드포인트에 대한 통합 테스트(Integration Test)를 작성합니다. 특히, 다수의 비동기 요청을 발생시켜 동시성 제어(분산 락)가 완벽하게 작동하는지 검증하는 스트레스 테스트 시나리오를 포함합니다.
- **데이터베이스 롤백 테스트:** 테스트 실행 시마다 임시 데이터베이스 세션을 생성하고, 테스트 종료 후 자동으로 롤백(Rollback)되도록 구성하여 멱등성(API Idempotency)이 보장된 테스트 환경을 구축합니다.

## 9. 인프라 및 배포 아키텍처 (Infrastructure & CI/CD)
- **컨테이너화 (Docker):** 프론트엔드, 백엔드, DB, Redis를 독립적으로 실행할 수 있도록 `docker-compose.yml`을 구성합니다.
- **CI/CD 파이프라인 (GitHub Actions):** main 브랜치 Push 시 자동 테스트(CI) 후 클라우드 서버에 자동 배포(CD)되도록 파이프라인을 구축합니다.
