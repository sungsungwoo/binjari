# 📝 빈자리(Binjari.com) 핵심 유스케이스 및 사용자 시나리오

## 문서 목적

본 문서는 빈자리(Binjari.com)의 MVP 범위 내 핵심 사용자 흐름과 시스템 동작을 정의합니다.
특히 예약 서비스의 핵심 리스크인 **중복 예약 방지, 실시간 상태 반영, 호스트 운영 정책 적용, 타임존 기준 정책 검증**이 실제 사용자 흐름 안에서 어떻게 동작하는지 명확히 설명하는 것을 목표로 합니다.

---

## 시나리오 1. [호스트] 예약 페이지 개설 및 슬롯 오픈 (온보딩)

호스트가 처음 가입한 뒤 자신의 예약 페이지를 생성하고, 운영 규칙과 예외 일정을 설정하여 예약 가능한 슬롯을 오픈하는 과정입니다.

* **액터(Actor):** 호스트 (`HOST` 권한 보유자)
* **사전 조건:** 호스트가 회원가입 및 로그인을 완료한 상태이며, 아직 생성된 예약 페이지가 없거나 새로운 예약 페이지를 추가하려는 상태
* **후행 결과:** 공개 가능한 예약 페이지가 생성되고, 예약 가능한 슬롯이 캘린더에 노출됨

### 단계별 흐름 (Main Flow)

1. 호스트가 대시보드 또는 예약 페이지 목록 화면에서 `[새 예약 페이지 만들기]` 버튼을 클릭합니다.
2. 호스트가 기본 설정 정보를 입력합니다.

   * 서비스명: `1:1 진로 상담`
   * URL 식별자(slug): `career-consulting-30min`
   * 호스트 타임존: `Asia/Seoul`
   * 슬롯 단위: `30분`
   * 승인 방식: `수동 승인(MANUAL)`
3. 시스템은 입력값을 검증한 뒤 `host_settings`를 생성합니다.
4. 호스트가 운영 시간 규칙을 설정합니다.

   * 월~금 `09:00~18:00` 오픈
   * `12:00~13:00` 휴게시간
5. 시스템은 `schedule_rules`를 저장합니다.
6. 호스트가 예외 일정을 설정합니다.

   * 다음 주 수요일: `DAY_OFF`
   * 사유: `개인 사정`
7. 시스템은 `schedule_overrides`를 저장합니다.
8. 호스트가 `[향후 30일 치 슬롯 생성]` 버튼을 클릭합니다.
9. 시스템은 운영 규칙과 예외 일정을 계산하여, 휴게시간과 휴무일을 제외한 슬롯을 30분 단위로 일괄 생성합니다.
10. 생성된 슬롯은 `slots` 테이블에 `OPEN` 상태로 저장됩니다.
11. 시스템은 호스트에게 공개용 예약 페이지 URL을 제공합니다.

* 예: `/book/career-consulting-30min`

### 예외 흐름 (Alternate Flow)

* **중복 slug 입력**

  * 동일한 `slug`가 이미 존재하면 시스템은 저장을 거부하고, 다른 URL 식별자를 입력하도록 안내합니다.
* **운영 규칙 오류**

  * 종료 시간이 시작 시간보다 빠르거나, 동일 요일에 시간이 겹치는 규칙이 입력되면 저장을 차단합니다.
* **중복 슬롯 생성 요청**

  * 동일 `host_setting_id`, 동일 `start_time`의 슬롯이 이미 존재하는 경우 시스템은 중복 생성을 허용하지 않습니다.
  * 이미 존재하는 슬롯은 건너뛰거나 정책에 따라 업데이트하지 않고 유지합니다.

## 시나리오 2. [예약자] 비로그인 상태에서 예약 시도

비로그인 사용자가 공개 예약 페이지에 진입하여 슬롯을 선택하고, 로그인 후 기존 맥락을 유지한 채 예약을 이어가는 과정입니다.

* **액터(Actor):** 비로그인 사용자
* **사전 조건:** 공개 예약 페이지 URL에 접속 가능한 상태
* **후행 결과:** 로그인 완료 후 원래 보던 예약 페이지로 복귀하여 예약을 계속 진행할 수 있음

### 단계별 흐름 (Main Flow)

1. 사용자가 공개 예약 페이지(`/book/{slug}`)에 접속합니다.
2. 사용자가 날짜를 선택하고 예약 가능한 슬롯 목록을 확인합니다.
3. 사용자가 특정 슬롯을 클릭합니다.
4. 시스템은 사용자의 로그인 여부를 확인합니다.
5. 사용자가 비로그인 상태이면 로그인 화면으로 이동시킵니다.
6. 사용자가 이메일 로그인 또는 구글 OAuth 로그인을 완료합니다.
7. 시스템은 사용자를 원래 보던 예약 페이지로 다시 이동시키고, 가능하면 직전에 선택한 날짜와 슬롯 정보를 복원합니다.
8. 사용자는 이어서 예약 신청을 진행합니다.

### 예외 흐름 (Alternate Flow)

* **로그인 실패**

  * 인증 실패 시 에러 메시지를 표시하고 로그인 화면에 머무릅니다.
* **복귀 시 슬롯 상태 변경**

  * 로그인 후 복귀했을 때 선택했던 슬롯이 이미 다른 사용자에 의해 점유되었으면, 시스템은 해당 슬롯이 더 이상 예약 불가함을 안내합니다.

## 시나리오 3. [예약자] 실시간 예약 신청 (수동 승인형)

예약자가 수동 승인 방식의 예약 페이지에서 특정 슬롯을 선점하고 예약 요청을 생성하는 과정입니다.

* **액터(Actor):** 로그인된 예약자
* **사전 조건:** 호스트의 예약 페이지에 접속해 있고, 해당 슬롯이 현재 예약 가능한 상태
* **후행 결과:** 예약 요청이 `PENDING` 상태로 생성되고, 호스트에게 새 예약 요청 알림이 전달됨

### 단계별 흐름 (Main Flow)

1. 예약자가 공개 예약 페이지(`/book/{slug}`)에서 특정 날짜를 클릭합니다.
2. 시스템은 해당 날짜의 `OPEN` 상태 슬롯 목록을 표시합니다.
3. 예약자가 `오후 2:00` 슬롯을 클릭합니다.
4. 시스템은 Redis에 해당 슬롯에 대한 5분짜리 임시 선점(Hold)을 생성합니다.
5. 예약 페이지 내부에 예약 신청 모달 또는 Bottom Sheet가 열리고, `04:59` 타이머가 표시됩니다.
6. 예약자가 요청 사항을 입력하고 `[예약 확정하기]` 버튼을 클릭합니다.
7. 요청에는 `Idempotency-Key`가 포함됩니다.
8. 시스템은 데이터베이스 트랜잭션을 시작하고, 해당 슬롯에 배타적 잠금(Pessimistic Lock)을 겁니다.
9. 시스템은 `bookings`에 `PENDING` 상태의 예약 요청을 생성합니다.
10. 시스템은 해당 슬롯의 `slots.status`를 `BOOKED`로 변경합니다.
11. 시스템은 트랜잭션을 커밋합니다.
12. 예약자에게 `"예약 요청이 완료되었습니다. 호스트의 승인을 기다려주세요."` 메시지를 표시합니다.
13. 시스템은 웹소켓을 통해 호스트 화면에 `"새로운 예약 요청이 들어왔습니다."` 알림을 전송합니다.

### 예외 흐름 (Alternate Flow)

* **타임아웃**

  * 사용자가 5분 안에 제출하지 않으면 Redis Hold가 만료됩니다.
  * 예약 신청 모달은 닫히고, 슬롯은 다른 사용자가 다시 선택할 수 있게 됩니다.
* **동시 클릭(경합)**

  * 다른 사용자가 먼저 Redis Hold를 획득한 상태라면, 시스템은 `"다른 사용자가 해당 시간대 예약을 진행 중입니다. 잠시 후 다시 시도해 주세요."` 메시지를 표시합니다.
* **중복 요청(Idempotency)**

  * 사용자가 네트워크 지연으로 버튼을 여러 번 눌러도 동일 `Idempotency-Key`가 감지되면, 시스템은 기존 처리 결과를 그대로 반환합니다.
* **최종 DB 검증 실패**

  * Redis Hold는 있었지만 DB 확정 시점에 이미 활성 예약이 존재하면, 시스템은 예약 요청 생성을 거부하고 슬롯이 마감되었음을 안내합니다.

## 시나리오 4. [예약자] 실시간 예약 신청 (자동 승인형)

예약자가 자동 승인 방식의 예약 페이지에서 슬롯을 선택하고 즉시 예약이 확정되는 과정입니다.

* **액터(Actor):** 로그인된 예약자
* **사전 조건:** `approval_type = AUTO`로 설정된 예약 페이지에서 예약 가능한 슬롯이 존재함
* **후행 결과:** 예약이 즉시 `CONFIRMED` 상태로 확정되고, 슬롯이 비활성화됨

### 단계별 흐름 (Main Flow)

1. 예약자가 공개 예약 페이지에서 특정 날짜와 슬롯을 선택합니다.
2. 시스템은 Redis Hold를 생성하고 예약 신청 모달 또는 Bottom Sheet를 엽니다.
3. 예약자가 `[예약 확정하기]` 버튼을 클릭합니다.
4. 시스템은 데이터베이스 트랜잭션을 시작하고 해당 슬롯에 배타적 잠금을 겁니다.
5. 시스템은 `bookings`에 `CONFIRMED` 상태의 예약을 생성합니다.
6. 시스템은 `slots.status`를 `BOOKED`로 변경합니다.
7. 시스템은 트랜잭션을 커밋합니다.
8. 예약자에게 `"예약이 확정되었습니다."` 메시지를 표시합니다.
9. 시스템은 호스트에게 `"새 예약이 확정되었습니다."` 알림을 보냅니다.
10. 다른 사용자가 동일 캘린더를 보고 있었다면, 해당 슬롯은 실시간으로 비활성화됩니다.

### 예외 흐름 (Alternate Flow)

* 수동 승인형과 동일하게 Redis Hold 만료, 동시 클릭, Idempotency 중복 요청, 최종 DB 검증 실패 케이스가 적용됩니다.

## 시나리오 5. [호스트] 예약 수동 승인 처리

호스트가 대기 중인 예약 요청을 확인하고 승인하는 과정입니다.

* **액터(Actor):** 호스트
* **사전 조건:** `PENDING` 상태의 예약 요청이 존재함
* **후행 결과:** 예약이 `CONFIRMED` 상태가 되고, 예약자와 다른 사용자에게 상태 변경이 반영됨

### 단계별 흐름 (Main Flow)

1. 호스트가 대시보드 또는 예약 요청 목록에서 `PENDING` 상태 예약을 확인합니다.
2. 호스트가 예약 상세 화면으로 이동하여 요청 내용을 검토합니다.
3. 호스트가 `[승인]` 버튼을 클릭합니다.
4. 시스템은 트랜잭션을 시작합니다.
5. 시스템은 `bookings.status`를 `CONFIRMED`로 변경합니다.
6. 시스템은 해당 슬롯의 `slots.status`가 `BOOKED` 상태임을 유지합니다.
7. 시스템은 트랜잭션을 커밋합니다.
8. 호스트 화면의 예약 리스트에서 해당 건이 `확정됨`으로 표시됩니다.
9. 시스템은 예약자에게 `"예약이 확정되었습니다."` 실시간 알림을 전송합니다.
10. 동일 캘린더를 보고 있는 다른 사용자 화면에서도 해당 슬롯은 계속 비활성 상태로 유지됩니다.

### 예외 흐름 (Alternate Flow)

* **이미 처리된 예약**

  * 예약이 이미 `CONFIRMED`, `REJECTED`, `CANCELLED` 상태로 변경되었다면, 시스템은 중복 승인을 허용하지 않습니다.
* **웹소켓 알림 실패**

  * 실시간 알림 전송에 실패하더라도 DB 상태 변경은 유지되며, 예약자는 이후 목록 재조회 시 최신 상태를 확인할 수 있습니다.

## 시나리오 6. [호스트] 예약 거절 처리

호스트가 대기 중인 예약 요청을 거절하고 슬롯을 다시 오픈하는 과정입니다.

* **액터(Actor):** 호스트
* **사전 조건:** `PENDING` 상태의 예약 요청이 존재함
* **후행 결과:** 예약 요청은 `REJECTED`가 되고 슬롯은 다시 예약 가능 상태로 전환됨

### 단계별 흐름 (Main Flow)

1. 호스트가 예약 상세 화면에서 `[거절]` 버튼을 클릭합니다.
2. 호스트가 거절 사유를 입력합니다.

   * 예: `"해당 시간 긴급 회의"`
3. 시스템은 트랜잭션을 시작합니다.
4. 시스템은 `bookings.status`를 `REJECTED`로 변경하고 거절 사유를 저장합니다.
5. 시스템은 `slots.status`를 `OPEN`으로 되돌립니다.
6. 시스템은 트랜잭션을 커밋합니다.
7. 예약자에게 `"예약 요청이 거절되었습니다."` 알림을 전송합니다.
8. 다른 예약자가 해당 캘린더를 보고 있었다면, 실시간 이벤트를 통해 슬롯이 다시 예약 가능 상태로 변경됩니다.

### 예외 흐름 (Alternate Flow)

* **이미 처리된 예약**

  * 예약이 이미 확정되었거나 취소되었으면 거절 처리를 허용하지 않습니다.

## 시나리오 7. [예약자] 예약 취소 및 슬롯 반환

예약자가 확정된 예약을 취소하고, 해당 슬롯이 다시 오픈되는 과정입니다.

* **액터(Actor):** 예약자
* **사전 조건:** 예약자가 취소 가능한 예약 건을 보유하고 있음
* **후행 결과:** 예약 상태가 `CANCELLED`로 바뀌고 슬롯이 다시 `OPEN` 상태가 됨

### 단계별 흐름 (Main Flow)

1. 예약자가 `[내 예약 목록]` 화면에 진입합니다.
2. 취소하려는 예약의 상세 화면으로 이동합니다.
3. 예약자가 `[예약 취소]` 버튼을 클릭합니다.
4. 시스템은 현재 시간이 예약 시작 시각보다 `cancel_deadline_minutes_before` 이상 이전인지, 호스트 타임존 기준으로 검증합니다.
5. 검증에 성공하면 시스템은 트랜잭션을 시작합니다.
6. 시스템은 `bookings.status`를 `CANCELLED`로 변경합니다.
7. 시스템은 해당 슬롯의 `slots.status`를 `OPEN`으로 되돌립니다.
8. 시스템은 트랜잭션을 커밋합니다.
9. 예약자에게 `"예약이 성공적으로 취소되었습니다."` 메시지를 표시합니다.
10. 시스템은 호스트에게 `"예약이 취소되어 슬롯이 다시 오픈되었습니다."` 알림을 전송합니다.

### 예외 흐름 (Alternate Flow)

* **취소 가능 시간 경과**

  * 현재 시간이 취소 마감 시각을 지난 경우, 시스템은 취소를 거부하고 `"예약 취소 가능 시간이 지났습니다."` 메시지를 표시합니다.
* **취소 불가능한 상태**

  * 이미 `CANCELLED`, `REJECTED`, `COMPLETED`, `NO_SHOW` 상태인 예약은 다시 취소할 수 없습니다.

## 시나리오 8. [시스템 공통] 실시간 상태 동기화 실패 시 복구

웹소켓 연결이 불안정하거나 끊어진 상황에서도 사용자가 최종 상태를 확인할 수 있도록 복구하는 과정입니다.

* **액터(Actor):** 시스템, 예약자, 호스트
* **사전 조건:** 상태 변경은 DB에 정상 반영되었으나, 웹소켓 이벤트 수신이 실패했거나 연결이 끊긴 상태
* **후행 결과:** 사용자가 목록 재조회 또는 화면 재진입 시 최신 상태를 확인할 수 있음

### 단계별 흐름 (Main Flow)

1. 예약 승인, 거절, 취소 등 상태 변경 이벤트가 DB에 정상 반영됩니다.
2. 웹소켓 연결 상태 문제로 일부 사용자에게 실시간 알림이 도달하지 않을 수 있습니다.
3. 사용자가 내 예약 목록, 예약 요청 목록, 예약 상세 화면에 재진입하거나 새로고침합니다.
4. 시스템은 API 재조회 결과를 기준으로 최신 상태를 다시 렌더링합니다.
5. 사용자는 실시간 알림을 놓쳤더라도 최종 상태를 정확히 확인할 수 있습니다.

## 공통 비즈니스 규칙 요약

1. **최종 예약 무결성은 DB가 보장한다.**

   * 활성 예약(`PENDING`, `CONFIRMED`)은 동일 슬롯에 최대 1건만 허용한다.

2. **임시 선점은 Redis가 담당한다.**

   * 사용자가 슬롯을 누르는 순간 Redis Hold를 생성하고, TTL 만료 시 자동 해제한다.

3. **예약 페이지의 정책 검증 기준은 호스트 타임존이다.**

   * 예약 가능 시점
   * 취소 가능 시점
   * 운영 시간 및 예외 일정 계산

4. **캘린더 노출 상태와 예약 상태는 역할이 다르다.**

   * `slots.status`는 현재 예약 가능 여부를 나타낸다.
   * `bookings.status`는 예약 요청의 실제 비즈니스 상태를 나타낸다.

5. **실시간 알림은 보조 채널이고, 최종 진실 공급원(Source of Truth)은 DB이다.**

   * 웹소켓 실패 시에도 목록 재조회로 최종 상태를 복구할 수 있다.
