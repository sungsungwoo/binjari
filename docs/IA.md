# 빈자리(Binjari.com) IA 문서

## 1. IA 설계 원칙

* MVP 범위에 포함된 기능만 화면으로 구성한다.
* 예약자와 호스트의 흐름을 분리하되, 공통 진입점은 홈/랜딩 페이지로 통합한다.
* 실시간 알림은 별도 페이지가 아니라 공통 UI 컴포넌트로 처리한다.
* 공개 예약 페이지는 `host_settings` 1건을 기준으로 노출한다. 랜딩의 탐색 그리드는 `GET /api/v1/public/marketplace/booking-pages`로, `is_active`·`is_listed`가 모두 true인 페이지만 조회한다.  

### 역할·권한 정책 (요약)

상세 정의·API 매핑은 [`docs/인증인가.md`](./인증인가.md) **§2.1**을 따른다.

| 역할 | 요약 |
|------|------|
| **Guest** | 비로그인. 공개 예약 페이지·슬롯 **조회**만 가능. **예약 불가** — 시도 시 로그인/회원가입 유도. |
| **Member** | 로그인한 일반 사용자. 내 예약, 공개 페이지 예약·취소·상태 확인. **예약 페이지 생성·호스트 운영 불가.** 회원가입은 **Admin 승인 없이 자동 완료**. |
| **Host** | `HOST` 역할. 예약 페이지 생성·수정, 슬롯·예외·승인/거절·통계 등. **로그인 후 기본 진입은 예약 페이지 목록**(`/host/services` 등). |
| **Admin** | `ADMIN` 역할. 가입 요청·계정 **승인/반려/정지** 등 운영(표준 Member 가입은 자동 승인이므로 별도 큐와 구분). |

## 2. 사이트맵

프론트 색·타이포 토큰: [`docs/DESIGN_TOKENS.md`](./DESIGN_TOKENS.md) · 구현: `frontend/` (Vite + React).

```text
/
├─ /                          홈 / 랜딩
├─ /auth/login                로그인
├─ /auth/signup               회원가입
├─ /book/{slug}              공개 예약 페이지(캘린더 + 슬롯 조회 + 예약 신청/확인 모달 + 예약 완료 상태)
  ├─ calendar view
  ├─ slot selection
  ├─ booking confirmation modal / bottom sheet
  └─ booking completion state
├─ /me/bookings               내 예약 목록 (Member 로그인 후 기본 진입)
├─ /me/bookings/{bookingId}   예약 상세
├─ /admin                     운영자 홈 (ADMIN, 가입 심사·상태 변경 예정)
└─ /host
   ├─ /dashboard              호스트 홈(페이지 0·1·N 분기: 1개면 해당 페이지 대시보드로 리다이렉트)
   ├─ /services               예약 페이지 목록(필터·카드·페이지네이션)
   ├─ /services/new           1단계: 초안 생성 후 `/services/{id}/setup` 마법사
   ├─ /services/{id}/dashboard   단일 예약 페이지 운영 허브
   ├─ /services/{id}/setup       2~4단계 마법사(규칙·슬롯·예외) + 활성화 요약
   ├─ /services/{id}/rules       운영 규칙(직접 편집)
   ├─ /services/{id}/overrides   예외 일정
   ├─ /services/{id}/slots       슬롯 생성·목록
   ├─ /bookings               예약 요청 목록(`?hostSettingId=` 로 페이지별 필터)
   ├─ /bookings/{bookingId}   예약 상세 / 승인·거절
   └─ /analytics              통계 대시보드(선택)
```

## 4. 역할별 사용자 흐름

### 4.1 Guest / Member (예약자)

* **Guest:** 홈/랜딩 또는 직접 링크 → `/book/{slug}`에서 월 이동·슬롯 **조회**까지 가능. 슬롯을 선택해 예약하려 하면 **로그인 또는 회원가입**으로 유도(토큰 없으면 Hold/예약 API 불가).
* **Member:** 회원가입은 **Admin 승인 없이 자동 완료** 후 로그인. 홈/랜딩 → 예약 페이지 → 날짜·슬롯 선택 → 예약 신청/확인(모달 또는 Bottom Sheet) → 예약 완료 → 내 예약 목록 → 예약 상세/취소. **호스트 전용 메뉴·예약 페이지 만들기**는 `HOST` 없이는 제공하지 않거나 진입 불가.

예약 신청/확인과 예약 완료는 별도 라우트로 분리하지 않고 `/book/{slug}` 내부 모달·Bottom Sheet·인라인 상태로 처리한다.

### 4.2 Host 플로우

로그인 → **예약 페이지 목록**(`/host/services`)을 기본 진입으로 하고, 필요 시 대시보드 분기(`/host/dashboard`) → 예약 페이지 생성 / 초기 설정 → 운영 규칙 → 예외 일정 → 슬롯 관리 → 예약 요청 목록 → 예약 상세/승인·거절 → 통계 대시보드

호스트가 먼저 `host_settings`를 만든 뒤 `schedule_rules`, `schedule_overrides`, `slots`, `bookings`를 관리하는 구조와 맞다. 최초 생성 시 title, `slot_duration_mins`, `approval_type` 등 기본 정책 입력이 선행된다.

### 4.3 Admin (운영자)

가입 요청·계정 목록 조회, **승인·반려·정지** 등 처리(구현·화면 범위는 로드맵에 따름). **일반 Member 회원가입**은 자동 활성이므로 Admin 큐와 구분한다. 상세는 [`docs/인증인가.md`](./인증인가.md) 참고.

## 5. 공통 UI / 시스템 요소

| 요소        | 위치                 | 설명                                        |
| --------- | ------------------ | ----------------------------------------- |
| 실시간 알림 UI | 전역 헤더/레이아웃         | 예약 승인, 거절, 취소, 새 예약 요청 등을 Toast 또는 배지로 표시 |
| 로딩 / 스켈레톤 | 예약 페이지, 목록 화면      | 월별 캘린더 조회, 예약 목록 조회 시 사용                  |

실시간 알림은 WebSocket 기반으로 제공하고, 예약 승인·거절·취소와 같은 상태 변경 이벤트를 사용자 또는 호스트에게 즉시 전달하도록 PRD에 정의돼 있으므로 별도 페이지가 아니라 전역 UI 요소로 처리하는 것이 맞습니다.

## 7. IA 설계 메모

* 공개 예약 페이지는 `host_settings.slug`를 기준으로 노출하며, 외부 공유 링크와 사용자 진입 경로는 `/book/{slug}`를 사용합니다.
* 내부 관리용 API 및 데이터 참조에는 `hostSettingId`(UUID)를 사용합니다.
* `예약 신청 / 확인`과 `예약 완료`는 별도 페이지가 아니라 `/book/{slug}` 내부 모달/하단 패널 및 인라인 상태 영역으로 처리합니다.
* `예약 요청 목록`과 `예약 상세 / 승인·거절`은 MVP 이후 하나의 분할 화면으로 통합할 여지도 있습니다.
* `실시간 알림 UI`는 페이지가 아니라 레이아웃 레벨 공통 컴포넌트로 정의하는 것이 적절합니다.

## 8. 최종 화면 수 요약

* 공개/인증: 3개 (홈, 로그인, 회원가입)
* 예약자: 5개
* 호스트: 8개 (대시보드, 페이지 목록, 페이지 생성, 규칙, 예외, 슬롯, 예약 목록, 예약 상세, 통계 - 표 기준 9개 항목이나, 맥락상 8~9개)
* 공통 UI: 1개

**총 17개 구성**
단, `실시간 알림 UI`는 별도 페이지가 아니므로 실제 라우트 기준으로는 **16개 화면**입니다.
