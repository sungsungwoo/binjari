# 🛡️ 빈자리(Binjari.com) 인증·인가 설계 (Auth & Security Architecture)

API 응답의 `error_code`·HTTP 상태 전체 목록은 [`docs/API_ERRORS.md`](./API_ERRORS.md)와 `backend/app/core/error_codes.py`를 참고한다.

## 1. 인증(Authentication) 설계: JWT 기반 무상태(Stateless) 세션

빈자리(Binjari.com)는 분산 서버 환경(Scale-out)에서의 확장성과 API 서버의 상태 비저장성을 확보하기 위해, 서버 메모리를 점유하는 세션(Session) 방식 대신 **JWT(JSON Web Token) 기반의 무상태 인증 구조**를 채택합니다.

핵심 원칙은 다음과 같습니다.

* **Access Token**은 짧은 수명의 인증 수단으로 사용한다.
* **Refresh Token**은 Access Token 재발급 전용으로만 사용한다.
* 인증 정보는 서버가 아닌 **토큰 서명 검증**을 통해 처리하되, 장기 토큰은 서버 측 저장소에서 관리한다.
* 최종 보안성은 **짧은 Access Token 수명 + Refresh Token Rotation + 안전한 쿠키 정책**으로 확보한다.

---

## 1.1 토큰 구조 및 생명주기 (Token Lifecycle)

### 1) Access Token (단기 토큰)

* **수명:** 30분 ~ 1시간
* **역할:** 실제 API 요청 시 `Authorization: Bearer <access_token>` 헤더에 담아 전송하며, 사용자의 인증 및 기본 권한 검증 수단으로 사용합니다.
* **특징:** 서버는 Access Token의 서명과 만료 시각을 검증하여 인증을 처리합니다.

#### 권장 Payload 구성

* `sub`: 사용자 식별자 (`user_id`)
* `email`: 사용자 이메일
* `roles`: 사용자 역할 목록 (예: `["HOST"]`, `["HOST", "ADMIN"]`)
* `iat`: 발급 시각
* `exp`: 만료 시각
* `jti`: 토큰 고유 식별자
* `iss`: 토큰 발급자
* `aud`: 토큰 대상

#### 설계 원칙

* Access Token은 **빠른 인증/인가 처리용 캐시성 정보**로 간주합니다.
* 역할 변경이 즉시 반영되지 않을 수 있으므로, Access Token은 **짧은 TTL**을 유지합니다.
* 민감한 기능(예: Admin API, 중요 설정 변경)은 필요 시 DB 재검증을 수행할 수 있습니다.

---

### 2) Refresh Token (장기 토큰)

* **수명:** 14일
* **역할:** Access Token이 만료되었을 때, 사용자가 다시 로그인하지 않고 새로운 Access Token을 발급받기 위한 용도로만 사용합니다.
* **사용 엔드포인트:** `POST /api/v1/auth/refresh`
* `/api/v1/auth/refresh`는 request body 없이 **HttpOnly Secure Cookie**에 저장된 Refresh Token으로만 동작합니다.

#### 저장 및 보안 정책

* Refresh Token은 **평문으로 저장하지 않고**, DB 또는 Redis에 **해시값으로 저장**합니다.
* 클라이언트에는 **HttpOnly + Secure 쿠키**로 전달합니다.
* 로그인/회원가입/OAuth 로그인 성공 응답에서는 Refresh Token을 response body에 포함하지 않고, `Set-Cookie` 헤더로만 전달합니다.
* Refresh Token은 **Rotation(회전)** 전략을 적용합니다.

#### Rotation 정책

* Refresh 요청이 성공하면:

  1. 기존 Refresh Token은 폐기
  2. 새 Refresh Token 발급
  3. 새 Refresh Token 해시를 서버 저장소에 저장
* 이미 폐기된 Refresh Token이 다시 사용되면, **토큰 탈취 또는 재사용 공격**으로 간주하고 해당 세션 또는 토큰 패밀리를 무효화합니다.

## 1.2 소셜 로그인 (Google OAuth 2.0) 흐름

빈자리는 Google 소셜 로그인에 대해 **Authorization Code Grant** 방식을 사용합니다.
보안이 취약한 Implicit Flow는 사용하지 않습니다.

### 권장 흐름

1. 프론트엔드가 구글 로그인 요청을 시작합니다.
2. 사용자는 구글 로그인/동의 화면을 거칩니다.
3. 구글은 인가 코드(Authorization Code)를 반환합니다.
4. 백엔드는 서버 간 통신(Server-to-Server)으로 구글에 접근하여 인가 코드를 검증하고 사용자 정보를 조회합니다.
5. 구글 사용자 정보(이메일, 프로필)를 기반으로 내부 사용자 계정을 조회/생성합니다.
6. 백엔드는 자체 **Access Token / Refresh Token**을 발급합니다.

### 보안 보강

* 프론트엔드가 인가 코드를 수신하는 구조일 경우, **PKCE(Proof Key for Code Exchange)** 를 함께 적용하여 인가 코드 탈취 위험을 줄입니다.
* OAuth 로그인 성공 후에도 최종 인증 수단은 구글 토큰이 아니라 **빈자리 자체 JWT** 입니다.

#### 백엔드 환경 변수 (구현 기준)

| 변수 | 설명 |
|------|------|
| `GOOGLE_OAUTH_CLIENT_ID` | Google Cloud Console OAuth 클라이언트 ID |
| `GOOGLE_OAUTH_CLIENT_SECRET` | 클라이언트 보안 비밀 |
| `GOOGLE_OAUTH_REDIRECT_URI` | 승인 후 콜백 URL (예: `http://127.0.0.1:8000/api/v1/auth/google/callback`, 콘솔에 동일 등록) |
| `FRONTEND_OAUTH_SUCCESS_URL` | 로그인 완료 후 브라우저 리다이렉트 URL (Refresh 쿠키는 API 도메인에 `Set-Cookie`) |

콜백 URL에 `response_mode=json`을 붙이면 JSON 응답(`AuthResponse` 형태)과 함께 Refresh 쿠키를 내려줍니다 (SPA·모바일 디버깅용).

## 1.3 로그아웃 및 세션 폐기

로그아웃은 단순히 클라이언트에서 토큰을 지우는 것으로 끝나지 않고, 서버 측 세션 무효화까지 포함해야 합니다.

### 로그아웃 처리 원칙

* 로그아웃 시 현재 Refresh Token 세션을 서버 저장소에서 폐기합니다.
* 클라이언트에 저장된 Refresh Token 쿠키를 삭제합니다.
* Access Token은 짧은 TTL을 통해 자연 만료되도록 처리합니다.

### 선택적 보안 강화

* Redis를 활용하여 **사용자 ID당 하나의 Refresh Token만 유효**하게 관리할 수 있습니다.
* 다른 기기에서 로그인 시 기존 세션을 강제로 만료시키는 정책도 선택적으로 적용할 수 있습니다.

## 2. 인가(Authorization) 설계: RBAC (역할 기반 접근 제어)

빈자리는 `roles`, `user_roles` 테이블 기반의 **RBAC(Role-Based Access Control)** 구조를 사용합니다.
사용자는 하나 이상의 역할을 가질 수 있으며, 예를 들어 한 사용자가 `HOST`이면서 동시에 `ADMIN`일 수 있습니다.

FastAPI에서는 `Depends()` 기반의 권한 가드를 활용하여 라우터 단위 인가를 구현합니다.

---

## 2.1 권한 레벨 및 가드 정책

### 1) Public API

* 토큰 없이 누구나 접근 가능 (**Guest**)
* 적용 예:

  * 로그인 / 회원가입
  * 공개 예약 페이지 조회
  * 예약 가능 슬롯 조회

Guest는 위 조회까지 가능하며, **슬롯 임시 선점·예약 생성** 등은 User API이므로 로그인이 필요하다. 클라이언트는 예약 시도 시 로그인·회원가입으로 유도한다.

### 2) User API

* 유효한 Access Token이 있는 **Member**(및 Host가 예약자 기능을 쓰는 경우 포함)
* 적용 예:

  * 내 예약 목록 조회
  * 예약 상세 조회
  * 예약 신청
  * 예약 취소

### 3) Host API

* Access Token의 `roles` 배열에 `"HOST"`가 포함된 사용자만 접근 가능
* 적용 예:

  * 예약 페이지 생성 / 수정
  * 운영 시간 설정
  * 예외 일정 설정
  * 슬롯 생성
  * 예약 승인 / 거절
  * 호스트 통계 조회

### 4) Admin API

* Access Token의 `roles` 배열에 `"ADMIN"`이 포함된 사용자만 접근 가능
* 적용 예:

  * 전체 사용자 조회
  * 가입 요청·계정 심사(승인·반려·정지) 등 운영 API
  * 시스템 설정 변경
  * 백오피스 기능

### 역할·권한 상세 (Guest / Member / Host / Admin)

제품·인가 기준으로 역할을 아래와 같이 정의한다. (DB의 `roles` 이름은 `HOST`, `ADMIN` 등 **문자열**로 유지하며, **Member**는 `user_roles`에 별도 행이 없어도 “로그인한 일반 사용자”로 해석한다.)

#### Guest (비로그인)

* 공개 예약 페이지·마켓플레이스·슬롯 **조회** 등 **1) Public API**만 이용 가능하다.
* **예약 불가**: 슬롯 선점(Hold)·예약 생성·내 예약 등 User 이상 API는 호출할 수 없다.
* UI에서는 예약 시도 시 **로그인 또는 회원가입**으로 유도한다.

#### Member (일반 사용자, 로그인)

* 로그인한 사용자로서 **2) User API**를 이용한다: 공개 예약 페이지 둘러보기, 예약 생성·취소, 내 예약 조회·상태 확인 등.
* **예약 페이지(Host) 생성·운영**은 불가 — **3) Host API**·`/host/*` 화면은 `HOST` 역할이 있어야 한다.
* **회원가입(이메일·비밀번호 또는 OAuth 최초 가입)은 Admin 승인 없이 자동 완료**된다. 가입 직후 활성 계정으로 로그인·User API 이용이 가능하다.
* `Member`와 **Admin 승인 큐**의 관계: 표준 일반 가입은 위와 같이 **즉시 활성**이며, Admin이 검토하는 것은 **심사가 필요한 별도 가입 유형**(예: 호스트 신청, 초대·제한 가입) 또는 **기존 계정 정지/재활성** 등 운영 정책으로 구분한다(구현 시 스키마·플로우로 반영).

#### Host (호스트 권한)

* Access Token의 `roles`에 **`"HOST"`** 포함.
* **3) Host API** 이용: 예약 페이지 생성·수정, 운영 규칙·예외 일정, 슬롯 관리, 예약 승인/거절, 호스트 통계 등.
* **로그인 후 기본 진입 화면**(호스트로 로그인했을 때의 홈)은 **예약 페이지 목록**(`/host/services` 등)을 기준으로 한다(랜딩 `/`가 아닌 호스트 허브).
* Host 사용자도 다른 호스트의 공개 페이지에 대해서는 **Member와 동일하게 예약**할 수 있다(제품 정책상 허용 시; 구현은 User API 동일).

#### Admin (운영자)

* Access Token의 `roles`에 **`"ADMIN"`** 포함.
* **4) Admin API** 및 운영자 전용 기능: 사용자·가입 요청 조회, **승인·반려·정지** 등 처리.
* **로그인 허용 조건**: 계정이 **정지·반려 등으로 로그인 불가** 상태가 아니어야 한다. 표준 **Member 자동 가입**으로 생성된 활성 계정은 즉시 로그인 가능하며, Admin은 **추가 심사 대상 가입** 또는 **정지/재활성** 등을 다룬다.

---

### API 레벨과 역할 매핑 요약

| API 레벨 | 대표 역할 | 비고 |
|----------|-----------|------|
| Public | Guest | 조회 위주; 예약·선점 제외 |
| User | Member (+ Host가 예약자로 행동) | Bearer 필수 |
| Host | `HOST` | 소유권 검증 별도 |
| Admin | `ADMIN` | Admin API 전용 |

## 2.2 역할 검증과 리소스 소유권 검증 분리

인가 설계에서 중요한 점은 **역할(Role) 검증과 리소스 소유권(Ownership) 검증은 별개**라는 점입니다.

예를 들어:

* 사용자가 `HOST` 역할을 가지고 있더라도
* 다른 호스트의 `host_settings`, `bookings`, `slots`를 수정할 수 있어서는 안 됩니다.

### 따라서 `/host/*` API는 최소 두 단계 검증이 필요합니다.

1. **역할 검증:** 요청자가 `HOST` 권한을 가진 사용자인가?
2. **소유권 검증:** 요청 대상 리소스가 해당 호스트 본인 소유인가?

### 적용 예

* `PATCH /host/booking-pages/{id}`
* `POST /host/booking-pages/{id}/toggle-listed`
* `POST /host/bookings/{bookingId}/approve`
* `POST /host/slots/{slotId}/block`

위 API는 모두 단순히 `HOST` 역할만 확인해서는 안 되고,
해당 예약 페이지/예약/슬롯이 요청자 본인의 리소스인지 검증해야 합니다.

---

## 2.3 인가 검증 최적화

API 요청마다 `user_roles`를 DB에서 조회하면 병목이 발생할 수 있습니다.
이를 줄이기 위해 **로그인 시점에 사용자의 역할을 DB에서 한 번만 읽어와 Access Token Payload에 주입**합니다.

예:

```json id="6n7ap5"
{
  "sub": "user-uuid",
  "roles": ["HOST"]
}
```

### 장점

* API 요청마다 DB 조회 없이 빠른 인가 처리 가능
* FastAPI `Depends()` 기반 가드에서 즉시 판별 가능

### 주의점

* 토큰 발급 이후 역할이 바뀌어도 Access Token 만료 전까지는 반영이 지연될 수 있음
* 이를 보완하기 위해:

  * Access Token 수명을 짧게 유지
  * 중요 기능은 DB 재검증 가능
  * 필요 시 `token_version` 또는 `role_version` 기반 무효화 전략을 고려

## 3. CORS 및 쿠키 보안 정책 (Cross-Domain Security)

빈자리는 프론트엔드(`binjari.com`)와 API 서버(`api.binjari.com`)가 분리된 구조를 전제로 보안 정책을 설계합니다.

---

## 3.1 CORS (Cross-Origin Resource Sharing) 설정

FastAPI의 `CORSMiddleware`를 엄격하게 설정하여 허용되지 않은 출처의 API 호출을 차단합니다.

### 정책

* `allow_origins`

  * 와일드카드(`*`) 사용 금지
  * 허용 출처만 명시
  * 예:

    * `https://binjari.com`
    * `http://localhost:5173`
* `allow_credentials=True`

  * Refresh Token 쿠키가 정상적으로 전달되도록 설정
* `allow_methods`

  * 필요한 메서드만 허용 (`GET`, `POST`, `PATCH`, `DELETE` 등)
* `allow_headers`

  * 필요한 헤더만 허용
  * 예:

    * `Authorization`
    * `Content-Type`
    * `Idempotency-Key`

---

## 3.2 토큰 저장 및 탈취 방지 전략 (XSS / CSRF 방어)

### 1) Access Token 저장 위치

* Access Token은 **프론트엔드 메모리(React State, Zustand 등)** 에 저장합니다.
* **LocalStorage / SessionStorage는 기본 저장소로 사용하지 않습니다.**

  * 이유: XSS 발생 시 토큰 탈취 위험이 크기 때문입니다.

### 2) Refresh Token 저장 위치

* Refresh Token은 **HttpOnly Secure Cookie** 로만 저장합니다.
* JavaScript에서 직접 접근할 수 없으므로 XSS 공격에 더 안전합니다.

### 쿠키 설정

* `HttpOnly`

  * `document.cookie`에서 접근 불가
* `Secure`

  * HTTPS 환경에서만 전송
* `SameSite=Lax` 또는 `Strict`

  * 크로스 사이트 요청에서 쿠키 전송 제한
* 필요 시 `Path`, `Domain`, `Max-Age`를 명시적으로 설정

---

## 3.3 CSRF 방어 전략

Refresh Token이 쿠키 기반으로 자동 전송되는 구조에서는 **CSRF 방어가 반드시 필요**합니다.

### 적용 대상

* `POST /api/v1/auth/refresh`
* `POST /api/v1/auth/logout`
* 기타 쿠키 기반 인증이 동작하는 엔드포인트

### 방어 전략

* `SameSite=Lax` 또는 `Strict` 쿠키 설정
* **Origin / Referer 검증**
* 필요 시 CSRF Token(Double Submit Cookie 포함) 추가 적용

즉, SameSite만으로 끝내지 않고 **Origin 검증을 함께 적용하는 것**을 기본 원칙으로 합니다.

## 4. WebSocket 인증 및 실시간 채널 보안

빈자리는 예약 승인/거절/취소 및 슬롯 상태 변경을 실시간으로 전달하기 위해 WebSocket을 사용합니다.
따라서 WebSocket 연결 역시 별도의 인증 정책이 필요합니다.

### 정책

* WebSocket 연결 시 Access Token 검증을 수행합니다.
* 토큰 전달은 query string 대신 `Sec-WebSocket-Protocol: bearer, <access_token>` 또는 짧은 수명의 WS 전용 티켓 방식을 사용합니다.
* 토큰 만료, 서명 불일치, 권한 부족 시 연결을 거부합니다.
* 호스트 전용 이벤트 채널은 `HOST` 권한을 가진 사용자만 구독 가능해야 합니다.
* 연결 후에도 필요 시 heartbeat / reconnect 전략을 사용합니다.

### 보안 원칙

* 실시간 알림은 **보조 채널**이고, 최종 상태의 진실 공급원(Source of Truth)은 DB입니다.
* WebSocket 메시지 수신 실패 시에도 사용자는 목록 재조회/API 재호출로 최신 상태를 확인할 수 있어야 합니다.

## 5.1 Rate Limiting

다음 엔드포인트에는 남용 방지를 위한 요청 제한을 적용합니다.

* 로그인
* 회원가입
* 비밀번호 재설정 요청
* Refresh Token 재발급
* OAuth 콜백
* 중요 액션 API(예약 생성, 승인/거절 등)

### 목적

* 브루트포스 공격 방지
* 토큰 남용 방지
* 자동화 공격 완화

---

## 5.2 에러 응답 규격화

에러는 프론트엔드가 일관되게 처리할 수 있도록 규격화된 JSON 형태로 응답합니다.

### 예시

```json id="oi2tyi"
{
  "success": false,
  "error_code": "INVALID_TOKEN",
  "message": "유효하지 않은 토큰입니다."
}
```

### 대표 상태 코드

* `401 Unauthorized`

  * 토큰 없음
  * 토큰 만료
  * 서명 불일치
  * Refresh Token 무효
* `403 Forbidden`

  * 인증은 되었으나 필요한 역할이 없음
  * 또는 해당 리소스의 소유자가 아님
* `409 Conflict`

  * Refresh Token 재사용 감지
  * 중복 로그인 제약 위반
* `429 Too Many Requests`

  * Rate Limit 초과

## 6. 인증·인가 아키텍처 요약

1. **인증은 JWT 기반 무상태 구조를 사용한다.**

   * Access Token은 짧게, Refresh Token은 길게 운용한다.

2. **Refresh Token은 서버 저장소에서 해시화 관리하며 Rotation을 적용한다.**

   * 재사용 감지 시 세션을 무효화한다.

3. **Access Token은 프론트엔드 메모리에 저장하고, Refresh Token은 HttpOnly Secure Cookie에 저장한다.**

   * LocalStorage는 기본 저장소로 사용하지 않는다.

4. **인가에는 RBAC를 사용한다.**

   * `HOST`, `ADMIN` 등 역할 기반 접근 제어를 적용한다.

5. **Host API는 역할 검증과 리소스 소유권 검증을 모두 수행한다.**

6. **쿠키 기반 인증 엔드포인트에는 SameSite + Origin/Referer 검증으로 CSRF를 방어한다.**

7. **WebSocket도 Access Token 기반으로 인증한다.**

   * 실시간 채널은 보조 수단이고, DB가 최종 상태 기준이다.

8. **모든 보안 민감 엔드포인트에는 Rate Limiting과 규격화된 예외 처리를 적용한다.**
